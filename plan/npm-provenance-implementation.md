## What is NPM Provenance?

NPM Provenance is a security feature that creates a **verifiable link** between a published npm package and its source code repository. It uses [SLSA (Supply-chain Levels for Software Artifacts)](https://slsa.dev/) attestations signed by [Sigstore](https://www.sigstore.dev/) to provide cryptographic proof of:

- **Where** the package was built (source repository and commit SHA)
- **How** the package was built (CI/CD workflow and build instructions)
- **Who** triggered the build (the CI/CD identity)

### Two Types of Attestations

1. **Provenance Attestation** - Links the package to its source code and build instructions
2. **Publish Attestation** - Generated by the registry when published by an authorized user

When a package is published with provenance, it is:
- Signed by Sigstore public good servers
- Logged in a public transparency ledger
- Displayed with a provenance badge on npmjs.com

## Why Implement Provenance?

### Security Benefits

| Benefit | Description |
|---------|-------------|
| **Supply Chain Security** | Users can verify the package came from the claimed source |
| **Tamper Detection** | Any modification to the package or provenance is detectable |
| **Build Transparency** | The exact build instructions are publicly auditable |
| **Credential Compromise Mitigation** | Attackers can't publish fake versions without leaving a trail |

### Real-World Attack Prevention

Notable npm supply chain attacks that provenance helps prevent:
- **UAParser.js** (2021) - Compromised credentials used to publish malicious versions
- **Command-Option-Argument** (2022) - Malicious code injected via hijacked account
- **rc package** (2018) - Credential-based attack on popular dependency

With provenance, even if credentials are compromised, the malicious package won't have valid provenance linking it to the legitimate repository.

## Implementation Options

### Option A: Traditional Provenance (with NPM_TOKEN)

**Best for:** Projects that want to keep using npm tokens but add provenance verification.

**Requirements:**
- npm CLI version 9.5.0+
- GitHub Actions with cloud-hosted runners
- `id-token: write` permission
- `NPM_TOKEN` secret configured

**Pros:**
- Minimal changes to existing workflow
- Still works with existing token-based infrastructure

**Cons:**
- Token still needs to be managed and rotated
- Token can still be compromised

---

### Option B: Trusted Publishing (Recommended)

**Best for:** Maximum security with no token management.

> 📢 **Generally Available as of July 31, 2025** - npm trusted publishing with OIDC is now GA!

**Requirements:**
- npm CLI version 11.5.1+
- GitHub Actions with cloud-hosted runners
- `id-token: write` permission
- Trusted publisher configured on npmjs.com

**Pros:**
- **No long-lived tokens** - Uses short-lived OIDC credentials
- **Automatic provenance** - No `--provenance` flag needed
- **Zero token exposure risk** - Nothing to leak or compromise
- **No rotation needed** - Credentials are ephemeral

**Cons:**
- Requires npm CLI 11.5.1+ (newer than default Node.js npm)
- Self-hosted runners not yet supported
- Only one trusted publisher per package

## Step-by-Step Implementation

### Option A: Traditional Provenance Implementation

#### Step 1: Update the GitHub Actions Workflow

Modify `.github/workflows/release.yml`:

```yaml
name: Automated Release

on:
  push:
    branches:
      - main

jobs:
  release:
    if: contains(github.event.head_commit.message, '[patch]') || contains(github.event.head_commit.message, '[minor]') || contains(github.event.head_commit.message, '[major]')
    runs-on: ubuntu-latest
    
    # ADD THIS: Required permissions for provenance
    permissions:
      contents: write      # For creating tags and releases
      id-token: write      # Required for npm provenance

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://registry.npmjs.org'

      - name: Install dependencies
        run: npm ci

      - name: Determine version bump type
        id: bump-type
        run: |
          if [[ "${{ github.event.head_commit.message }}" == *"[major]"* ]]; then
            echo "type=major" >> $GITHUB_OUTPUT
          elif [[ "${{ github.event.head_commit.message }}" == *"[minor]"* ]]; then
            echo "type=minor" >> $GITHUB_OUTPUT
          else
            echo "type=patch" >> $GITHUB_OUTPUT
          fi

      - name: Get current version
        id: current-version
        run: |
          CURRENT_VERSION=$(node -p "require('./package.json').version")
          echo "version=$CURRENT_VERSION" >> $GITHUB_OUTPUT

      - name: Calculate new version
        id: new-version
        run: |
          CURRENT="${{ steps.current-version.outputs.version }}"
          TYPE="${{ steps.bump-type.outputs.type }}"
          IFS='.' read -ra VERSION_PARTS <<< "$CURRENT"
          MAJOR=${VERSION_PARTS[0]}
          MINOR=${VERSION_PARTS[1]}
          PATCH=${VERSION_PARTS[2]}
          case $TYPE in
            major) NEW_VERSION="$((MAJOR + 1)).0.0" ;;
            minor) NEW_VERSION="$MAJOR.$((MINOR + 1)).0" ;;
            patch) NEW_VERSION="$MAJOR.$MINOR.$((PATCH + 1))" ;;
          esac
          echo "version=$NEW_VERSION" >> $GITHUB_OUTPUT

      - name: Update package.json version
        run: npm version ${{ steps.new-version.outputs.version }} --no-git-tag-version

      - name: Build package
        run: npm run package

      # UPDATED: Add --provenance flag
      - name: Publish to npm with provenance
        run: npm publish --provenance --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

      - name: Create git tag
        run: |
          git config --local user.email "action@github.com"
          git config --local user.name "GitHub Action"
          git tag "v${{ steps.new-version.outputs.version }}"
          git push origin "v${{ steps.new-version.outputs.version }}"

      - name: Create GitHub release
        uses: actions/create-release@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          tag_name: v${{ steps.new-version.outputs.version }}
          release_name: Release v${{ steps.new-version.outputs.version }}
          body: |
            Automated release v${{ steps.new-version.outputs.version }}
            
            ✅ Published with npm provenance
            
            Changes in this release:
            - ${{ github.event.head_commit.message }}
```

**Key Changes:**
1. Added `permissions.id-token: write`
2. Added `--provenance` flag to `npm publish`
3. Added `--access public` (required for first publish with provenance)

### Option B: Trusted Publishing Implementation (Recommended)

1. Go to [npmjs.com](https://www.npmjs.com/) and sign in
2. Navigate to your package: `sveltekit-openapi-generator`
3. Click **Settings** → **Trusted Publisher**
4. Click **GitHub Actions** button
5. Fill in the configuration:
   - **Organization or user:** `Michael-Obele`
   - **Repository:** `sveltekit-api-gen`
   - **Workflow filename:** `release.yml`
   - **Environment name:** (leave empty unless using GitHub Environments)

After verifying trusted publishing works:

1. Go to package settings on npmjs.com
2. Navigate to **Settings** → **Publishing access**
3. Select **"Require two-factor authentication and disallow tokens"**
4. Click **Update Package Settings**
5. Revoke the old `NPM_TOKEN` from your npm account
6. Remove the `NPM_TOKEN` secret from GitHub repository settings

## Troubleshooting

### Error: "Unable to authenticate"

**Causes:**
1. Workflow filename doesn't match exactly (case-sensitive, must include `.yml`)
2. Using self-hosted runners (not supported)
3. Missing `id-token: write` permission

**Solution:**
- Verify the workflow filename matches exactly what's configured on npmjs.com
- Ensure you're using `runs-on: ubuntu-latest` (GitHub-hosted runner)
- Add `permissions.id-token: write` to the job

### Error: "Provenance generation failed"

**Causes:**
1. npm CLI version is too old (need 9.5.0+ for provenance, 11.5.1+ for trusted publishing)
2. Private repository (provenance not supported for private repos)

**Solution:**
```yaml
- name: Update npm
  run: npm install -g npm@latest
```

### Provenance Badge Not Showing

**Causes:**
1. Package was published from a private repository
2. The `--provenance` flag wasn't used (for traditional method)
3. Registry hasn't updated yet (wait a few minutes)

**Solution:**
- Provenance only works with **public repositories** publishing **public packages**
- Verify the publish command includes `--provenance`

### "repository" Field Mismatch

**Cause:** The repository URL in package.json doesn't match where you're publishing from.

**Solution:** Ensure exact match in `package.json`:
```json
{
  "repository": {
    "type": "git",
    "url": "git+https://github.com/Michael-Obele/sveltekit-api-gen.git"
  }
}
```

## SLSA Provenance Format

For reference, here's what the provenance attestation looks like:

```yaml
_type: https://in-toto.io/Statement/v0.1
subject:
  - name: pkg:npm/sveltekit-openapi-generator@0.1.3
    digest:
      sha512: <package-hash>
predicateType: https://slsa.dev/provenance/v0.2
predicate:
  buildType: https://github.com/npm/cli/gha/v2
  builder:
    id: https://github.com/actions/runner
  invocation:
    configSource:
      uri: git+https://github.com/Michael-Obele/sveltekit-api-gen@refs/heads/main
      digest:
        sha1: <commit-sha>
      entryPoint: ".github/workflows/release.yml"
    environment:
      GITHUB_EVENT_NAME: push
      GITHUB_REF: refs/heads/main
      GITHUB_REPOSITORY: Michael-Obele/sveltekit-api-gen
      GITHUB_WORKFLOW_REF: Michael-Obele/sveltekit-api-gen/.github/workflows/release.yml@refs/heads/main
```

## FAQ: Can I Publish Locally with Provenance?

### Short Answer: No

**You cannot publish with npm provenance from your local machine.** This is by design, not a limitation.

### Why Local Provenance Isn't Possible

| Requirement | Local Machine | CI/CD (GitHub Actions) |
|-------------|---------------|------------------------|
| OIDC Identity Token | ❌ Not available | ✅ Provided by GitHub |
| Sigstore Signing | ❌ No trusted identity | ✅ Uses CI/CD identity |
| Verifiable Build Environment | ❌ Opaque to users | ✅ Publicly auditable |
| Reproducible Build Proof | ❌ No attestation | ✅ Logged in transparency ledger |

The whole point of provenance is to cryptographically prove that a package was built from a specific commit in a **trusted, auditable environment**. Your local machine doesn't have a trusted OIDC identity that npm/Sigstore will accept.

### What You CAN Do Locally

1. **Build the package** - `npm run package`
2. **Test the package** - `npm test`
3. **Dry-run publish** - `npm publish --dry-run`
4. **Version the package** - `npm version patch/minor/major`
5. **Commit and push** - to trigger CI/CD workflow

### Recommended Workflow

```
┌─────────────────────────────────────────────────────────────┐
│                     LOCAL DEVELOPMENT                        │
├─────────────────────────────────────────────────────────────┤
│  1. Make changes to your code                               │
│  2. Run tests: npm test                                     │
│  3. Build: npm run package                                  │
│  4. Verify: npm publish --dry-run                           │
│  5. Commit: git commit -m "feat: new feature [patch]"       │
│  6. Push: git push origin main                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   GITHUB ACTIONS (CI/CD)                     │
├─────────────────────────────────────────────────────────────┤
│  1. Detects [patch], [minor], or [major] in commit message  │
│  2. Bumps version in package.json                           │
│  3. Builds the package                                      │
│  4. Publishes with provenance (--provenance flag)           │
│  5. Creates git tag                                         │
│  6. Creates GitHub release                                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       NPM REGISTRY                           │
├─────────────────────────────────────────────────────────────┤
│  ✅ Package published with provenance badge                 │
│  ✅ Attestation logged in Sigstore transparency ledger      │
│  ✅ Users can verify package origin                         │
└─────────────────────────────────────────────────────────────┘
```

### Your Options

| Option | Provenance | How |
|--------|------------|-----|
| Publish locally | ❌ No | `npm publish` (no provenance badge) |
| Publish via CI/CD | ✅ Yes | Push with `[patch]`/`[minor]`/`[major]` in commit message |

### Why This Is Actually Better

Publishing via CI/CD with provenance is **more secure** than local publishing because:

1. **No credentials on your machine** - Your NPM token (or no token with trusted publishing) is only in GitHub Secrets
2. **Audit trail** - Every publish is tied to a specific commit and workflow run
3. **Reproducibility** - Anyone can see exactly how the package was built
4. **Protection against compromise** - Even if your laptop is compromised, attackers can't publish with valid provenance

## Summary

Implementing npm provenance significantly improves the security posture of your package by providing verifiable proof of its origin and build process. 

**Key Takeaway:** You must use CI/CD (GitHub Actions or GitLab CI/CD) to publish with provenance. Local publishing without provenance is still possible but not recommended.

**Recommended Approach:** Use **Trusted Publishing** (Option B) for maximum security benefits:
- No tokens to manage or rotate
- Automatic provenance generation
- Industry-standard OIDC authentication
- Eliminates credential compromise risks

The investment of ~30 minutes to set this up provides lasting security benefits for your package's users and the broader npm ecosystem.
